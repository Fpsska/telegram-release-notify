# Desktop UI для telegram-release-notify — дизайн

Дата: 2026-07-02
Статус: утверждён пользователем (брейншторм с визуальными макетами, см. `.superpowers/brainstorm/`)

## Цель

Десктоп-приложение (Windows, .exe для команды) поверх существующей логики: парсинг коммитов → JIRA (статусы, исполнители) → уведомление в Telegram. CLI-режим сохраняется без изменений интерфейса. Настройки из `.env` переезжают в UI.

## Ключевые решения

| Вопрос | Решение |
|---|---|
| Ввод коммитов | Вставка текста в textarea (как аргументы CLI), парсинг regex `[A-Z]+-\d+` |
| Дистрибуция | Standalone .exe (PyInstaller), команда без Python |
| Настройки | `%APPDATA%\release-notify\settings.json` — единый источник для UI и CLI; `.env` остаётся fallback-ом (CI/совместимость). UI пишет только в settings.json |
| Контроль | Предпросмотр: таблица тикетов с галочками + предпросмотр сообщения перед выполнением |
| Стек | pywebview + HTML/CSS/JS фронтенд, ядро — чистый Python |
| Layout | Мастер из трёх шагов: Ввод → Проверка → Результат |
| Тема | Тёмная (GitHub-подобная палитра: фон #0d1117/#1a1f27, акцент #2f81f7) |

## Архитектура

```
telegram-release-notify/
├── release_notify.py        # CLI — тонкая обёртка над core (обратная совместимость)
├── core/
│   ├── __init__.py
│   ├── config.py            # загрузка: %APPDATA%\release-notify\settings.json → .env fallback
│   ├── tickets.py           # extract_jira_tickets (regex)
│   ├── jira_client.py       # find_issues, смена статусов (BFS по workflow_matrix), исполнители
│   ├── telegram.py          # send_telegram + прокси
│   └── workflow_matrix.json # упаковывается в .exe как ресурс
├── app/
│   ├── main.py              # точка входа UI: окно pywebview + js_api мост
│   ├── api.py               # Python-класс, методы зовутся из JS
│   └── web/                 # index.html, style.css, app.js
├── requirements.txt         # + pywebview
└── build.spec               # PyInstaller: один .exe с иконкой
```

- **Ядро в `core/`**: убрать сайд-эффекты текущего `release_notify.py` (JIRA-подключение при импорте, `os.environ[...]` без default). Конфиг загружается явно, JIRA-клиент создаётся по запросу. CLI и UI зовут одни и те же функции.
- **CLI без изменений**: `python release_notify.py QA 26.1.0 7 "..."` работает как раньше.
- **JS ↔ Python**: pywebview `js_api`. Долгие операции — в потоке; прогресс пушится в JS через `window.evaluate_js`.

## UI: мастер из трёх шагов

Окно ~700×550, тёмная тема, шестерёнка настроек в шапке, индикатор шагов сверху.

**Шаг 1 «Ввод»**: окружение (dropdown со свободным вводом), релиз, RC, textarea для коммитов. «Далее» → `api.parse_and_fetch(commits)`: парсинг + загрузка тикетов из JIRA в фоне (спиннер). Результат: `{key, summary, type, status, reporter}` на тикет + список тикетов с ошибкой загрузки.

**Шаг 2 «Проверка»**: таблица тикетов с галочками (по умолчанию все включены; не найденные — серые, без галочки, с текстом ошибки). Чипы типа (Bug/Task) и статуса. Живой предпросмотр Telegram-сообщения, обновляется при снятии галочек. «Выполнить» → `api.execute(selected_keys, env, release, rc)`.

**Шаг 3 «Результат»**: живой лог (статусы → исполнители → отправка), построчный пуш из Python. Итог по каждому тикету: ✓ / ⚠ с причиной. Кнопки «Новый прогон» и «Копировать лог».

При старте с пустыми/невалидными настройками — сразу экран настроек с подсказкой.

## Экран настроек

Открывается по ⚙. Три группы:

1. **Telegram**: Bot Token (скрыт, 👁), Chat ID, прокси (опционально, hint с форматом). Кнопка «Проверить: отправить тест в чат». Бейдж статуса подключения.
2. **JIRA**: хост (без https://), логин, пароль (скрыт). Кнопка «Проверить подключение». Бейдж статуса.
3. **Команда QA**: тестировщики (логины через запятую; если репортер в списке — тикет назначается ему), QA-лид (получает остальные).

«Сохранить» → `%APPDATA%\release-notify\settings.json` (plaintext; шифрование секретов отклонено на брейншторме как избыточное).

## Обработка ошибок

- JIRA недоступна / 401 → баннер на шаге 1 «Проверь логин/пароль в настройках» + кнопка перехода в настройки.
- Тикет не найден → строка с ⚠ в таблице, остальные тикеты работают.
- Нет пути в workflow_matrix / transition недоступен → warning в лог, выполнение продолжается (сообщение в Telegram уходит) — текущее поведение.
- Telegram не отвечает → в лог текст сообщения + кнопка «Повторить отправку» (повторяется только отправка, статусы не трогаются).
- CLI: поведение и вывод не меняются.

## Тестирование

- Юнит-тесты: `core/tickets.py` (regex-извлечение), `core/config.py` (приоритет settings.json → .env), BFS-поиск пути в `core/jira_client.py`. JIRA мокается.
- Ручная проверка UI: полный прогон с тестовым Telegram-чатом.
- Сборка: `build.spec`; проверить, что `workflow_matrix.json` и `app/web/` включены в .exe ресурсами.

## Вне скоупа

- Git-интеграция (чтение git log) — отклонено, только вставка текста.
- Светлая тема / переключатель тем — отклонено, только тёмная.
- Windows Credential Manager для секретов — отклонено.
- macOS/Linux сборки — не требуются (команда на Windows).
